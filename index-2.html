  1. Open Close Principle

    04 October 2015

    I am still getting familiar with the SOLID principles and am trying to assess their value only when I see their value in bad code that I have written or problems that I come up against.

    But I think reading through them at this stage is a good exercise. The Open Close Principle is the next one I am looking at, which states, that software entities should be closed for modification but open for extension.

    That is, that it should be possible to make changes to the exisiting code and extend its behaviour without breaking the exisiting functionality of the code. Interfaces help with this as they allow you to extend the code more easily and help create compact classes that hold the required behaviour.

    I am using this principle to a certain degree without realising it, but will try and be more concious of when I am using it and make sure I do so when required.

  3. TDD

    04 October 2015

    I was talking to someone at a workshop who used the comparison of TDD with a religion. The reason given for this was that there is no definitive study that shows code written using TDD is better than that not using it. So it is mainly faith that causes people to use TDD. Now I have researched this a little and I know it is not true. There are scientific studies that prove the benefits.

    The idea of TDD is to write a test for some behaviour you expect, run the test to see it fail, write as little code as possible to see the test pass and then refactor if possible when the test is green. The next steps are to repeat this process until all the behaviour you want in your program has been tested effectively.

    I feel the benefit of using TDD even in the small katas and programs that I am writing as I learn more. So I am going to keep using tests before I write, even though I have some way to go in learning which tests to start with. Hopefully this will come with practice.

  4. Reviewing Code & Pull Requests

    04 October 2015

    Something I am working on is writing easier to read code. Previously I was so focused on getting my code to work as quickly as possible, I did not spend as much time as I should have naming as best as I could.

    I am also working on separating out methods that only handle one piece of behaviour. Some of the other apprentices and I are going to start reviewing each others code and makimng pull requests so the readability of the code is going to be increasingly important.

    Of all the various components of good clean code, naming is probably the most important so I am going to do my best to make mine as easy to read as possible.

  5. Code Retreat

    04 October 2015

    When I first heard about code retreats I thought a holiday or trip was involved at some point. I suppose that was the original idea, that people would go and meet each other at a specific location and code together.

    I took part in my first code retreat today and it was really fun! The retrospectives about what has been learnt is particularly interesting, but the part I enjoyed was working with people and seeing how they approach the specific problem, especially when it is so different to mine.

    I saw how some people communicate through narration, others let code do the talking and others ask questions. I was even exposed to a new language and had to be quiet for a full 20 minutes when trying to complete a specific task with my pair, which was a struggle.

    We had 20 minute iterations, or pairing sessions with a specific constraint and this made the challenges difficult, but it was a great way to get some exposure to a code retreat in a short space of time.

    I am looking forward to my next one!
