  1. UML Diagrams

    09 September 2015

    UML diagrams are commonly used when modelling for object oriented design programs. They are the graphical notation for drawing diagrams of software concepts.

    Composite relationships are shown in a UML diagram by an arrow with one end a point and the other a diamond.

    Inheritance is shown through a blank arrowhead and the arrows point in the direction that the inheritance flows from.

    Class icons are divided into three components, a class title, the variables and the functions and interfaces are shown by the <> notation.

    It is always a good idea to draw up a UML before starting to write a program, and adjust it as you go.

  2. Loops in Java

    08 September 2015

    Loops in Java are useful even in the most simple program and contain their contents within curly braces. The two most common ones I have used so far are the for loop;

    eg. for( int i = 0; i < 5; i++ )

    and the while loop;

    eg. while ( i < number.length )

    Keeping track of the variables and loops can again be clarified through the use of an instruction table.

    It is also possible to escape out of a loop if a certain condition is met using a label just before the beginning of the loop and a break after the end of the loop.

  3. The Single Responsibility Principle

    07 September 2015

    The single responsibility principle means that a class should only have one reason to change.

    The idea is that each class should only have one role or handle one piece of behaviour. This does not mean that each class should contain one method, but if when describing a class behaviour, if the word ‘and’ is used, it is quite likely that the class is doing too much and the additional behaviour should be split out.

    If the SRP is implemented well, then the design of the program will be more flexible and easier to change.

    This is because if a class changes then only the class that handles that behaviour will have to change and the code will not be too coupled–meaning that one small change that must be made leads to numerous other things being forced to change.

  4. Input/Output

    04 September 2015

    Everything has to be explicitly declared in Java. I am starting to get used to it, even though the format looks different, the general structure of a class is not too different to Ruby. However, in Java there are a lot more restrictions on what you can and cannot do compared to a dynamic language.

    In order to use Input Output in Java, the simplest way to do this is the InputStreamReader and OutputStreamReader.

    The sytnax for these is System.in and System.out.print() respectively, but the format of these has to be changed accordingly when a certain type of display is needed.

    It is also possible to use the Scanner class that parses primitive types and strings suing regular expressions.

    There are so many tools and libraries that you can import in java that help you, it can be difficult to figure out which ones are best to use, but I am sure this is something that comes with practice and familiarity with the language. At the moment, I am avoiding using these until I get comfortable with what the standard Java Library contains.

  5. Throwing Exceptions

    02 September 2015

    Throwing an exception is a way of handling an error that a method may produce for a specific reason that was expected by the programmer who wrote it.

    In Java, a complier will not let you compile a method if an exception is written that has not been handled or specifically declared.

    Even though it is possible to declare that the main method throws the exception, it is always nicer to pass the declaration to another method and print a message to the user after another method has caught the exception. It is also good practice to tell the user how to address the error if at all possible.
