  1. Creating Objects

    09 December 2015

    As Java documentation explains, a class is like a blueprint that can be used to create an object. Each time an object is created by calling new ObjectName an object constructed with the relevant constructors. 

    Square square = new Square(50,50);

    A class can have multiple constructors, and it is possible to create variations of an object by using these different constructors (if the constructors are different they must have different signatures. A constructor has the same name as the class and has no return type.

    
public class Shape {

    public Shape(int width, int height) {
        this.width = width;
        this.height = height;
    }

    public void areaOfSquare() {
        return width * height;
    }

  2. Writing tests

    09 December 2015

    Constructing tests and deciding where to start is difficult, especially at the very beginning. A useful place to start is the most basic use case, an empty array or string representing the final construction you want to start building. This is a good entry point and from there you can start building out what you want to construct step by step, closer to the final product.

    A good practice, that is difficult to adpot, is to think about what your code would ideally look like on execution and write a test that reflects this. This is of course more difficult than you would imagine. Most of the time, even before a test has been written you have some idea of how you want to di this in code. I think it is a good idea to try and detach and try on just focus on the tests and go from there. This is something I know I definitely need to work on.

  3. Leadership Workshop

    02 December 2015

    Today all the Apprentices and Craftsmen of the 8th Light London office attended a Leadership workshop and spent a few hours discussing what values we hold dear and which ones we would like to improve upon.

    There was quite a bit of introspection involved, which was good. Sometimes when you are so busy it is difficult to find the time to pause and reflect. I usually use the time I run and exercise to do this, but I have fallen out of practice with it and am not running as regularly as I used to. So it was nice to have time dedicated to think about what I would like to personally aim for.

    Something that came up for me quite frequently was quality, but really if you looked carefully at most of the results that came up frequently there was quite some room for interpretation. Quality to me can be applied to most areas of life. To me quality just means putting time and care into something, that leads to better results naturally. This can mean quality time with friends and family, quality in cooking a meal, quality in building a house, quality in a performance. It is again the aggregation of marginal gains and doing things with purpose. I am quite an excitable person, which definitely has its time and place but I would like to sometimes slow down and pay even more attention to detail which I am sure will help my work.

    So this is what I personally got out of the workshop, I am going to take time to pause and reflect more and strive for quality and consideration in everything (well most things anyway!) that I do.

  4. The Liskov Principle

    09 November 2015

    The Liskov Principle was formulated by Barbara Liskov and Jeannette Wing in 1994. It was described by them as:

    Let Φ(x) be a property provable about objects x of type T. Then Φ(y) should be true for objects y of type S where S is a subtype of T.

    This essentially means that subtypes of a an object may replace the object itself, without totally changing the functioning of a program.

    In java this is ususally implemented with the use of abstract classes and interfaces.

    The typical example for this is the square and the rectangle, which I think does not illustrate the example very well. A much better expalination was given by one of the craftsmen during a Zagaku this week. Here is the link: Liskov.

  5. 05 November 2015

    As my minesweeper program is getting bigger and I am starting to think about the different classes that could emerge from the existing ones it already has, it is becoming increasingly important to think about how they should be separated.

    I am currently reading Kent Beck’s Implementation Patterns and he describes the various categories of class separation and state. There are a lot of categories, and I think the only way to really fully understand their use is to actually use them in my code.

    The ones I have come across already are Interfaces, and Superclasses to some extent. He also talks about specialization and the types of questions you should be asking if you want to think about how to separate out classes. Some patterns I have not really used yet are the Implementor pattern and the Inner Class.

    Most of the state based patterns are concerned with how an object changes state. This is primarily done through the internals of the object, so all the methods it contains.

    I am going to try and notice when I use these patterns in my code and write up how I am trying to use them.
