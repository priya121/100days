  2. Polymorphism

    16 October 2015

    Polymorphism is one of the greatest advantages of OOP (object oriented desin). It basically means you can call different methods on an object and change its form.

    This allows you to create many categories of an object. In Java all objects are polymorphic, because they are all part of the class Object.

    A class can gave different underlying data, but can be presented in the same form as others due to polymorphism.

    Polymorphism is a real advantage to OOP and even though I was not sure what the definition was, I have been using it a lot already.

  3. Decorator Pattern

    07 October 2015

    I have heard about design patterns and their uses, how they are just tools that you can use to possibly improve your code. That they have their uses, but also their limitations.

    But I think it is important to be aware of them at the very least and then decide whether or not you want to or should use them.

    The first one I have come across at the classroom sessions we have daily is the Decorator Pattern that handles common instances by wrapping them within one another.

    This is done in Java through the use of interfaces which I have implemented in my Guess the Number game.

    This allows for the addition of new instances with slight variations that do not impact the original instance and so many different variations can be created easily and this makes extra requirements cleaner and easier to implement.

    I think this was a good design pattern to start with, I am going to work my way through the patterns and try and experiment with my code by adding them in.

  5. Using Fakes in Java

    04 October 2015

    I am starting to really see the benefit of using fakes in my code. Working through a Guess the Number game and creating a fake that simulates the input and intermediate output is useful when writing tests because you need to be able to test behaviour that is not certain and can change each time a program is run.

    The way this is done is through the use of an interface. An interface is essentially a class that contains the methods or behaviour that must be present for your class to work. The class that ultimately uses this behaviour is not aware of the behaviour of the object that implements the interface, provided that it contains the methods that the interface requires.

    In the tests you can then create class that implements the methods or behaviour the interface requires, but set what they return to a value that is hardcoded. Then the actual console implementation can be passed in when the actual game is run so that the Fakes hardcoded returns can be replaced with a user input and the correct output.

    Guess the number is a simple enough program to test this so this is a good way to start understanding IO and fakes in Java.
